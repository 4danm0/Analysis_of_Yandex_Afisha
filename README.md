## 📌 Описание проекта

Ежедневно **Яндекс Афиша** помогает тысячам людей находить интересные события в их городе — от громких театральных премьер до уютных концертов — и покупать на них билеты. Сервис выступает надёжным мостом между организаторами мероприятий, билетными операторами и зрителями.

В рамках данного проекта проведён глубокий исследовательский анализ данных (EDA) с целью изучения паттернов поведения пользователей и выявления признаков тех, кто с наибольшей вероятностью станет постоянным клиентом.

## 💼 Бизнес-контекст

Команда маркетинга Яндекс Афиши поставила амбициозную цель: **сместить фокус с разовых продаж на долгосрочное удержание клиентов** и увеличить долю повторных покупок.

**Ожидаемые результаты исследования помогут бизнесу:**

| Цель | Описание |
|------|----------|
| 🎯 Персонализировать опыт | Быстро выявлять перспективных клиентов и предлагать им эксклюзивные условия |
| 📈 Повысить ROI | Точнее настраивать рекламные кампании на аудиторию с высоким потенциалом возврата |
| 💰 Оптимизировать затраты | Грамотно распределять маркетинговый бюджет, не распыляясь на нецелевой трафик |
| 🔄 Увеличить LTV | Повысить общий уровень удержания и пожизненную ценность клиента |

## 📊 Данные

Данные выгружены из базы данных через SQL и содержат информацию о заказах пользователей:

| Столбец | Описание |
|---------|----------|
| `user_id` | Уникальный идентификатор пользователя |
| `device_type_canonical` | Тип устройства (`mobile` / `desktop`) |
| `order_id` | Уникальный идентификатор заказа |
| `order_dt` | Дата создания заказа |
| `order_ts` | Дата и время создания заказа |
| `currency_code` | Валюта оплаты |
| `revenue` | Выручка от заказа |
| `tickets_count` | Количество купленных билетов |
| `days_since_prev` | Количество дней от предыдущей покупки |
| `event_id` | Уникальный идентификатор мероприятия |
| `service_name` | Название билетного оператора |
| `event_type_main` | Основной тип мероприятия |
| `region_name` | Название региона проведения мероприятия |
| `city_name` | Название города проведения мероприятия |

**Объём данных:** ~290 000 строк до очистки.

## 🔄 Этапы выполнения

### 1. 📥 Загрузка данных и предобработка

- Выгрузка данных из PostgreSQL через `SQLAlchemy`
- Оценка корректности и объёма выгрузки
- Первичное ознакомление со структурой данных

### 2. 🔧 Предобработка данных

- **Конвертация валют:** выручка приведена к единой валюте — российскому рублю (данные в тенге переконвертированы по курсу 2024 года)
- **Типы данных:** `order_dt` и `order_ts` приведены к типу `datetime`
- **Обработка пропусков:** пропуски только в `days_since_prev` — ожидаемо для пользователей с одним заказом
- **Фильтрация выбросов:** по 99-му перцентилю для `revenue_rub` (удалено < 3% строк)
- **Удаление дублей:** удалены дублирующие заказы с разницей менее 2 минут (~6% строк)

### 3. 👤 Создание профиля пользователя

Для каждого пользователя рассчитаны агрегированные признаки:

- Дата первого и последнего заказа
- Устройство, регион, партнёр и жанр первого заказа
- Общее количество заказов
- Средняя выручка с одного заказа
- Среднее количество билетов в заказе
- Среднее время между заказами
- Бинарные флаги `is_two` (2+ заказа) и `is_five` (5+ заказов)

### 4. 🔍 Исследовательский анализ данных

#### 4.1. Признаки первого заказа и их связь с возвращением

- Распределение пользователей по типу мероприятия, устройству, региону, оператору
- Анализ возвратов по сегментам
- Проверка продуктовых гипотез

#### 4.2. Поведение через показатели выручки и состава заказа

- Сравнение распределений средней выручки между группами (1 заказ vs 2+)
- Анализ влияния среднего количества билетов на повторные покупки

#### 4.3. Временные характеристики и их влияние на удержание

- Влияние дня недели первой покупки на вероятность возврата
- Анализ среднего интервала между заказами (группы 2–4 и 5+)

#### 4.4. Корреляционный анализ (phi_k)

- Расчёт универсального коэффициента корреляции `phi_k` между признаками профиля и числом заказов
- Визуализация тепловой карты корреляций

## 📈 Ключевые результаты

### Проверка гипотез

| Гипотеза | Результат |
|----------|-----------|
| Пользователи со спортивных мероприятий возвращаются чаще, чем с концертов | ❌ Не подтвердилась |
| В регионах с большей активностью выше доля повторных заказов | ❌ Не подтвердилась |

### Факторы удержания

🔑 **Среднее количество билетов в заказе** — наиболее важный фактор:

| Сегмент | Доля повторных покупок |
|---------|----------------------|
| 1 билет (только на себя) | ~40% |
| 2–3 билета (на компанию) | ~74% |
| 5+ билетов | Высокая активность |

💰 **Ценовой фактор:**
- Пользователи с 1 заказом чаще покупают билеты по более низкой стоимости (диапазон 0–100 руб.)
- Возвращающиеся пользователи концентрируются в диапазоне 150–250 руб.

📅 **Временные паттерны:**
- День недели первого заказа **не влияет** на вероятность возврата
- Короткий интервал между заказами (10–15 дней) ассоциирован с переходом в сегмент 5+ заказов

## 💡 Рекомендации

### 🎯 Фокус на сегменте "2–3 билета"

Это самый перспективный сегмент по удержанию. Рекомендуется:
- Групповые предложения и механики «пригласи друга»
- Персональные подборки мероприятий для посещения вдвоём/втроём

### ⏰ Работа с риском длинных пауз

Настроить триггерные коммуникации после заказа:
- На **7–14 день** — рекомендации и персональные подборки
- На **21–30 день** — напоминания и специальные предложения

### 🏷️ Осторожнее со «сильной скидкой на первый заказ»

Альтернативные механики:
- Бонус на второй заказ
- Кэшбэк после первого заказа
- Скидка на повторную покупку

### 📊 Таргетинг на основе поведения, а не статики

Устройство, регион, тип события и сервис первого заказа дают мало сигнала. Для таргетинга лучше использовать:
- `avg_tickets_count` — среднее число билетов в заказе
- Признаки регулярности покупок
- Ценовой уровень заказов

## 🚀 Установка и запуск

### 1. Клонирование репозитория

```bash
git clone https://github.com/4danm0/Analysis_of_Yandex_Afisha.git
cd Analysis_of_Yandex_Afisha
```

### 2. Установка зависимостей

```bash
pip install -r requirements.txt
```

### 3. Настройка переменных окружения

Создайте файл `.env` в корне проекта и добавьте параметры подключения к базе данных:

```env
DB_HOST=your_host
DB_PORT=your_port
DB_NAME=data-analyst-afisha
DB_USER=your_user
DB_PASSWORD=your_password
```

### 4. Запуск ноутбука

```bash
jupyter notebook project_modul_2.ipynb
```
